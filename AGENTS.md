# AGENTS.md

このファイルは、このリポジトリで作業するAIエージェント向けのREADMEです。
人間向けの概要やセットアップ説明は `README.md`、プロダクト要件は
`docs/requirements.md` を参照してください。

## 1. プロジェクト概要

インターン生と企業をつなぐスカウトサービスのプロトタイプを開発します。

- フロントエンド: Next.js
- バックエンド: Rails（JSON API）
- データベース: PostgreSQL
- 必須機能: `docs/requirements.md` のMVPスコープを参照
- 要件の正: `docs/requirements.md`

要件にない仕様を推測だけで大きく拡張しないでください。不明点が実装結果や
データ設計を大きく左右する場合は、作業前に質問してください。

## 2. リポジトリ構成

現在の構成は次のとおりです。

```text
.
├── frontend/          # Next.jsアプリケーション
├── backend/           # Rails APIアプリケーション
├── docs/              # 要件・設計資料
├── .github/workflows/ # CI
├── compose.yaml       # Rails・PostgreSQL開発環境
├── AGENTS.md
└── README.md
```

実際の構成が変更された場合は、コードと同じ変更内でこの節も更新してください。

## 3. 作業開始時の確認

1. `git status --short` で既存の変更を確認する。
2. `README.md` と `docs/requirements.md` を読む。
3. 対象ディレクトリ内に、より深い階層の `AGENTS.md` がないか確認する。
4. 関連する既存コードとテストを読み、命名・設計パターンを合わせる。
5. 変更範囲を小さく定め、無関係な修正を混ぜない。

ユーザーや他のエージェントによる未コミット変更は保持してください。

## 4. セットアップ・ビルド・テスト

アプリケーション作成後は、実際に採用したパッケージマネージャーやスクリプトに
合わせてこの節を更新してください。ロックファイルに対応するツールを使い、別の
パッケージマネージャーのロックファイルを追加しないでください。

### フロントエンド

`frontend/` で実行します。npm以外のlockfileを追加しないでください。

```bash
npm ci
npm run dev
npm run lint
npm run typecheck
npm run test
npm run build
```

### バックエンド

ホストのRubyには依存せず、リポジトリ直下からDocker Composeで実行します。

```bash
docker compose build backend
docker compose up -d db
docker compose run --rm backend bin/rails db:prepare
docker compose up backend
docker compose run --rm -e RAILS_ENV=test \
  -e DATABASE_URL=postgresql://postgres:postgres@db:5432/intern_scout_test \
  backend bin/rails test
docker compose run --rm backend bin/rails zeitwerk:check
```

テストフレームワークはRails標準のMinitestを採用しています。

### 変更後の最低限の検証

- フロントエンド変更: lint、関連テスト、production build
- バックエンド変更: 関連テスト、全体テスト、`zeitwerk:check`
- DB変更: migrationのupとdown、または同等のロールバック可能性を確認
- API変更: フロントエンドとの型・パラメータ・レスポンス整合性を確認
- ドキュメントのみ: リンク、コマンド、ファイル名が実態と一致することを確認

環境上実行できない検証がある場合は、未実行の項目と理由を報告してください。

## 5. 開発方針・コード規約

### 共通

- 既存のformatter、linter、型チェック設定を優先する。
- 変更は要件を満たす最小単位にし、早すぎる抽象化を避ける。
- コメントは「何をしているか」より、判断理由や制約を書く。
- 認証情報、APIキー、個人情報をコード・fixture・ログへ含めない。
- 環境変数を追加したら `.env.example` とREADMEを更新する。
- 外部依存を追加する際は、必要性、保守状況、ライセンス、代替手段を確認する。

### Rails

- Rails標準の命名とディレクトリ構成に従う。
- controllerはHTTP処理に集中させ、複雑な業務ロジックを詰め込まない。
- 入力にはStrong Parametersを使用し、認可を必ずサーバー側で行う。
- migrationは原則として後から編集せず、新しいmigrationで変更する。
- N+1クエリ、無制限の一覧取得、機密項目のJSON露出を避ける。
- modelやAPIの振る舞いを変更するときはテストを先に追加または更新する。

### Next.js

- TypeScriptを使用し、`any` の追加を避ける。
- Server Componentを基本とし、ブラウザAPIや操作状態が必要な箇所だけ
  Client Componentにする。
- API通信処理、表示コンポーネント、フォーム状態を必要に応じて分離する。
- 認証・認可を画面の非表示だけに依存させない。
- loading、empty、errorの各状態を明示的に扱う。
- アクセシブルなHTMLを優先し、フォームにはlabelとエラー表示を設ける。

## 6. テスト方針

- 不具合修正では、可能な限り先に再現テストを追加する。
- 新機能では、正常系だけでなく認証・認可・入力エラーも検証する。
- 実装詳細ではなく、利用者から見た振る舞いをテストする。
- 外部サービスに依存するテストは境界でstubし、再現性を保つ。
- 重要導線である登録、プロフィール閲覧、メッセージ送信・閲覧を優先する。
- テストを削除・skipして失敗を回避しない。

## 7. API・データ変更のルール

- APIは `/api/v1` のようにバージョンを明示する方針とする。
- エラーレスポンスの形式とHTTPステータスを統一する。
- 一覧APIにはページネーションを設ける。
- 破壊的変更は避け、必要な場合は利用箇所を同じ変更内で更新する。
- 個人情報は必要最小限だけ保存・返却し、ログへ出力しない。
- メッセージは送信者と参加者の整合性を検証し、第三者から参照できないようにする。

## 8. Git・PRルール

- 1つのPRは1つの目的に絞る。
- ブランチ名は `feature/...`、`fix/...`、`docs/...` など目的が分かる名前にする。
- コミットメッセージは命令形または簡潔な変更内容で記述する。
- 自動生成物や大容量ファイルを、理由なくコミットしない。
- force push、履歴改変、他者の変更の取り消しを無断で行わない。
- PR本文には次を記載する。
  - 目的と背景
  - 主な変更点
  - 動作確認・テスト結果
  - UI変更時のスクリーンショット
  - DB・環境変数・APIへの影響
  - 残課題または対象外
- lint、テスト、buildが通る状態でレビューを依頼する。

## 9. ガードレール

次の操作は禁止、または明示的な承認を必要とします。

- `.env`、秘密鍵、トークンなどの秘密情報をコミットすること
- 本番データや実在人物の個人情報を開発・テストへ流用すること
- `git reset --hard`、強制push、無断のファイル一括削除
- 既存migrationの書き換えによる適用済みDB履歴の破壊
- 認証・認可・バリデーションを一時的に無効化したまま提出すること
- テスト失敗を、テスト削除やskipによって隠すこと
- 要件外の大規模リファクタリングを同じPRへ混ぜること
- 本番環境へのデプロイ、課金サービスの作成、外部へのメッセージ送信を
  明示的な依頼なしに行うこと

## 10. 完了条件

- `docs/requirements.md` の受け入れ条件を満たしている。
- 変更に対応するテストがあり、必要な検証が成功している。
- lint、型チェック、buildに新しいエラーがない。
- API、DB、環境変数、操作方法の変更が文書化されている。
- セキュリティと個人情報の観点で見直している。
- 未実行の検証、既知の制約、残課題を作業報告に明記している。

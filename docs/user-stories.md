# ユーザーストーリー

## 1. 目的

5日間のMVPで実現する利用者価値と受け入れ条件を、実装単位で整理する。
要件の正は `requirements.md` とし、本書は開発順序とテスト観点を補足する。

優先度はすべてMUSTとする。通知、未読、リアルタイム更新、選考管理などは
今回のユーザーストーリーへ含めない。

## 2. インターン生

### US-01 アカウント登録

インターン生として、メールアドレスとパスワードで登録したい。
プロフィールを作成し、企業からスカウトを受けられるようになるため。

受け入れ条件:

- 有効な入力でインターン生アカウントを作成できる。
- 重複メールアドレスや不正な入力は保存されず、理由が表示される。
- 登録直後の未完成プロフィールは企業向け一覧へ掲載されない。

### US-02 プロフィール登録・編集

インターン生として、自分のプロフィールを登録・編集したい。
経験や制作物、希望を企業へ伝えるため。

受け入れ条件:

- 表示名、学校名、学年、経験・制作物を必須入力できる。
- 希望職種と最大20件の技術スタックを任意入力できる。
- 必須項目が揃うと企業向け一覧へ掲載される。
- 他の利用者はプロフィールを編集できない。

### US-03 募集閲覧

インターン生として、公開中の募集一覧と詳細を閲覧したい。
興味のある企業と仕事を見つけるため。

受け入れ条件:

- 公開募集だけを一覧・詳細で閲覧できる。
- 募集タイトル、企業名、業務内容、技術スタック、勤務条件を確認できる。
- 非公開募集をURLまたはAPIから直接閲覧できない。

### US-04 募集応募

インターン生として、公開中の募集へ応募したい。
掲載企業とのコミュニケーションを始めるため。

受け入れ条件:

- 応募前に対象募集を確認できる。
- 応募すると、応募記録と企業との会話が作成または再利用される。
- 会話へ定型の応募メッセージが自動送信される。
- 同じ企業には1回だけ応募でき、別募集も応募済みと表示される。
- 失敗時に応募記録またはメッセージだけが残らない。

## 3. 企業担当者

### US-05 アカウント登録

企業担当者として、企業名、メールアドレス、パスワードで登録したい。
候補者の検索、スカウト、募集掲載を始めるため。

受け入れ条件:

- 有効な入力で企業担当者アカウントと企業情報を作成できる。
- 企業名を省略できない。
- MVPでは審査なしで登録できる。

### US-06 インターン生一覧・詳細

企業担当者として、掲載済みインターン生の一覧と詳細を閲覧したい。
候補者を比較し、連絡する相手を判断するため。

受け入れ条件:

- 掲載済みプロフィールだけを1ページ20人で閲覧できる。
- 一覧から詳細へ移動できる。
- 未ログイン利用者とインターン生には情報を開示しない。

### US-07 インターン生検索

企業担当者として、学校名、希望職種、技術スタックで絞り込みたい。
条件に合う候補者を効率よく見つけるため。

受け入れ条件:

- 各条件を部分一致で指定できる。
- 異なる種類の条件はANDで組み合わせる。
- 条件変更後も20人単位で表示する。
- 該当者がいない場合は空状態を表示する。

### US-08 スカウト送信

企業担当者として、インターン生へスカウトメッセージを送りたい。
候補者とのコミュニケーションを開始するため。

受け入れ条件:

- 1文字以上2,000文字以下のメッセージを送信できる。
- 同じ企業とインターン生の既存会話があれば再利用する。
- 既存会話がある学生の詳細ではスカウト入力欄を表示せず、会話を開ける。
- 空または2,000文字を超える本文は保存されない。
- 別企業の担当者は会話を閲覧できない。

### US-09 募集作成・編集

企業担当者として、自社の募集を作成・編集・公開停止したい。
インターン生へ仕事の機会を提示するため。

受け入れ条件:

- タイトル、業務内容、技術スタック、勤務条件を設定できる。
- 企業紹介とホームページURLを設定し、すべての自社募集へ共通表示できる。
- 非公開で保存し、公開または公開停止できる。
- 他社の募集は編集できない。
- 公開停止後も過去の応募と会話は残る。

## 4. 共通

### US-10 ログイン・ログアウト

利用者として、安全にログイン・ログアウトしたい。
自分に許可された情報だけを利用するため。

受け入れ条件:

- 正しい認証情報でログインできる。
- 誤った認証情報では利用者の存在を必要以上に開示しない。
- ログアウト後は認証が必要なAPIを利用できない。
- インターン生と企業担当者は許可された画面へ移動する。

### US-11 会話一覧・返信

利用者として、自分の会話とメッセージ履歴を確認して返信したい。
スカウトまたは応募後のやり取りを継続するため。

受け入れ条件:

- 自分が参加する会話だけを一覧表示する。
- 相手、最新メッセージ概要、更新日時を確認できる。
- 履歴を送信日時順に閲覧し、2,000文字以内で返信できる。
- 会話参加者以外からの閲覧・送信を拒否する。

### US-12 アカウント削除

利用者として、自分のアカウントを削除したい。
サービスの利用を終了できるようにするため。

受け入れ条件:

- 削除前に影響と、送信済みメッセージが相手へ残ることを確認できる。
- 削除後はログイン、掲載、応募、返信、新規送信ができない。
- 個人プロフィールと認証情報は削除または復元不能に匿名化される。
- 既存メッセージは相手に残り、送信者は「退会済みユーザー」と表示される。
- 企業退会時は公開募集がすべて非公開になる。

### US-13 アクセス制御

利用者として、自分に関係のない情報を他者に見られないようにしたい。
プロフィール、応募、会話の情報を安全に保つため。

受け入れ条件:

- Rails APIが認証、ユーザー種別、所有者、会話参加者を検証する。
- UIを経由せずURLやAPIを直接指定しても情報を取得できない。
- 権限不足とリソース不在から機密情報を推測しにくい応答にする。

## 5. 開発順序

1. US-01、US-05、US-10: 登録・認証
2. US-02、US-06、US-07: プロフィール・企業向け検索
3. US-08、US-11、US-13: スカウト・会話・認可
4. US-03、US-09、US-04: 募集・応募
5. US-12: 退会と匿名化

各段階で正常系だけでなく、入力エラー、認証、認可、空状態を検証する。

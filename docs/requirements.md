# インターンスカウトサービス 要件定義書

## 1. 文書情報

- ステータス: MVP要件確定
- 対象: 5日間で開発するプロトタイプ
- 最終更新日: 2026-07-04

## 2. 背景と目的

企業が自社に合うインターン生を見つけ、直接コミュニケーションを始められる
スカウトサービスを開発する。インターン生から企業への応募導線も用意し、企業発の
スカウトとインターン生発の応募を同じ会話機能へつなげる。

このプロトタイプでは、次の価値を検証する。

- インターン生がプロフィールを登録し、企業へ自分の情報を伝えられる。
- 企業が条件に合うインターン生を探し、スカウトできる。
- 企業が募集を掲載し、インターン生が閲覧・応募できる。
- スカウトまたは応募をきっかけに、双方がメッセージを継続できる。

## 3. 成功条件

デモ環境またはローカル環境で、次の主要導線を完了できること。

1. インターン生が登録し、プロフィールを完成させる。
2. 企業担当者が登録し、インターン生を検索してスカウトする。
3. インターン生がスカウトを確認して返信する。
4. 企業担当者が募集を作成して公開する。
5. インターン生が公開募集を閲覧して応募する。
6. 応募により企業との会話が作成または再利用され、自動メッセージが届く。
7. 利用者がアカウントを削除でき、相手側には匿名化された履歴が残る。
8. 無関係な利用者がプロフィールや会話へ不正アクセスできない。

## 4. スコープ

### 4.1 MVPに含める機能

- インターン生と企業担当者のアカウント登録
- ログイン・ログアウトとユーザー種別に応じた認可
- インターン生プロフィールの登録・編集・掲載
- 企業向けインターン生一覧・詳細
- 学校名、希望職種、技術スタックによる検索・絞り込み
- 企業とインターン生の1対1メッセージ
- 会話一覧とメッセージ履歴
- 企業による募集の作成・編集・公開停止
- インターン生による公開募集の一覧・詳細閲覧
- インターン生による募集への応募と応募メッセージの自動送信
- アカウント削除と共有メッセージ履歴の匿名化保持
- デモ用seedデータ
- 無料構成による公開デモ環境

### 4.2 MVPの対象外

- 決済
- 管理者向け管理画面
- 企業の本人確認・審査
- メール・プッシュ通知と未読管理
- ファイル添付とメッセージのリアルタイム配信
- 応募辞退、選考ステータス管理、応募者専用一覧
- 募集のお気に入り、募集検索、募集と通常スカウトの明示的な紐付け
- 予定調整とグループチャット
- ネイティブモバイルアプリ

## 5. 利用者と権限

### 5.1 インターン生

- 自分のアカウントとプロフィールを登録・編集できる。
- 公開中の募集を閲覧し、1企業につき1回応募できる。
- 自分が参加する会話だけを閲覧し、メッセージを送信できる。
- 企業向けのインターン生一覧・詳細は閲覧できない。
- 自分のアカウントを削除できる。

### 5.2 企業担当者

- 企業名、メールアドレス、パスワードで自己登録できる。
- 掲載済みインターン生の一覧と詳細を閲覧・検索できる。
- インターン生との会話を開始し、メッセージを送信できる。
- 自社の募集だけを作成・編集・公開停止できる。
- 自分が参加する会話と自社募集への応募だけを閲覧できる。
- 自分のアカウントを削除できる。

MVPでは1企業1担当者とし、同一企業の複数担当者には対応しない。企業登録は
プロトタイプ上では誰でも行えるものとし、審査は行わない。

## 6. 機能要件

### FR-01 アカウント登録

- 利用者はインターン生または企業担当者を選んで登録できる。
- 共通の必須項目はメールアドレスとパスワードとする。
- 企業担当者は企業名も必須とする。
- メールアドレスは重複登録できない。
- パスワードは安全にhash化し、平文では保存しない。
- 入力に不備がある場合、修正方法が分かるエラーを表示する。

### FR-02 認証・認可

- 登録済み利用者はログイン・ログアウトできる。
- 認証にはHttpOnly Cookieを用いたセッション方式を採用する。
- 未認証利用者は認証が必要な画面・APIを利用できない。
- ユーザー種別とリソースの所有・参加関係をRails API側で検証する。

### FR-03 インターン生プロフィール

インターン生は次の情報を登録・編集できる。

- 表示名（必須）
- 学校名（必須）
- 学年（必須）
- 経験・制作物（必須）
- 希望職種（任意）
- 技術スタック（任意、自由入力、最大20件）

学年は `1年`、`2年`、`3年`、`4年`、`5年`、`修士1年`、`修士2年`、
`博士課程`、`その他` から選択する。文字数上限は表示名50文字、学校名100文字、
経験・制作物1,000文字、希望職種100文字、技術スタック1件50文字とする。

登録直後のプロフィールは企業向け一覧へ掲載しない。必須項目がすべて揃った時点で
自動的に掲載する。実名、住所、電話番号などMVPに不要な個人情報は収集しない。

### FR-04 インターン生の一覧・詳細・検索

- ログイン済み企業担当者だけが掲載済みプロフィールを閲覧できる。
- 一覧は1ページ20人でページネーションする。
- 学校名、希望職種、技術スタックの部分一致で絞り込める。
- 複数種類の条件を指定した場合、すべての条件を満たす候補者を表示する。
- 該当者がいない場合は、空であることが分かる表示を行う。
- 未ログイン利用者と他のインターン生には一覧・詳細を公開しない。

### FR-05 会話の開始

- 企業担当者はインターン生詳細から会話を開始できる。
- 同じ企業とインターン生の組み合わせでは、既存の会話を再利用する。
- 既存の会話がある場合、インターン生詳細ではスカウト入力欄の代わりに会話への導線を表示する。
- インターン生は通常操作から未知の企業へ新規会話を開始できない。
- 公開募集への応募時だけ、応募先企業との会話を開始または再利用できる。

### FR-06 メッセージ

- 会話の参加者は1文字以上2,000文字以下のテキストを送信できる。
- メッセージには会話、送信者、本文、種類、送信日時を記録する。
- 通常メッセージと応募による自動メッセージを区別できる。
- 会話の参加者以外は履歴を閲覧・送信できない。
- メッセージは送信日時順に表示する。

### FR-07 会話一覧

- 利用者は自分が参加する会話だけを一覧で閲覧できる。
- 各会話には相手、最新メッセージの概要、更新日時を表示する。
- 会話を選択するとメッセージ履歴を閲覧できる。
- 退会済みの相手は「退会済みユーザー」と表示し、新たな送信を禁止する。

### FR-08 募集の作成・編集

- 企業担当者は自社募集を作成・編集・公開停止できる。
- 募集にはタイトル、業務内容、求める技術スタック、勤務条件を設定できる。
- 企業紹介と企業ホームページURLを企業共通の情報として設定できる。
- 募集状態は非公開または公開の2種類とする。
- 他社の募集は編集できない。

文字数上限はタイトル100文字、業務内容2,000文字、勤務条件1,000文字、
企業紹介2,000文字、企業ホームページURL2,048文字、技術スタック1件50文字・最大20件とする。
企業ホームページURLはHTTPまたはHTTPSだけを受け入れる。

### FR-09 募集の閲覧

- インターン生は公開中の募集一覧と詳細を閲覧できる。
- 募集には掲載企業名を表示する。
- 募集詳細には企業紹介と企業ホームページへのリンクを募集内容と分けて表示する。
- 非公開募集は、作成企業以外へ開示しない。

### FR-10 募集への応募

- インターン生は公開中の募集へ応募できる。
- 応募前に確認画面を表示する。
- 応募記録、会話の作成または再利用、自動メッセージ送信を一つのDB処理で行う。
- 自動メッセージは「募集『募集タイトル』に応募しました。プロフィールをご確認ください。」とする。
- 同じインターン生は同じ企業の別募集を含めて重複応募できない。
- 応募後は「応募済み」と表示する。
- 募集を非公開に変更しても、過去の応募と会話は保持する。

### FR-11 アカウント削除

- 利用者は確認画面を経て、自分のアカウントを削除できる。
- 削除後はログイン、掲載、応募、返信、新規メッセージ送信を禁止する。
- 認証情報と個人プロフィールを削除または復元不能な値へ匿名化する。
- 企業が退会した場合は自社募集をすべて非公開にする。
- 送信済みメッセージは相手側に残し、送信者を「退会済みユーザー」と表示する。
- 削除確認画面で、送信済みメッセージが相手に残ることを明示する。

## 8. データ要件（概念モデル）

### User

- 認証情報、ユーザー種別（intern / company）、退会状態を持つ。
- InternProfileまたはCompanyと1対1で関連する。
- 会話の参加者とメッセージの送信者はUserへ関連付ける。

### InternProfile

- インターン生Userに1対1で属する。
- 表示名、学校名、学年、経験・制作物、希望職種を持つ。
- 最大20件のTechnicalStackと関連する。

### Company

- 企業担当Userに1対1で属し、企業名を持つ。
- 複数のJobPostingを持つ。

### TechnicalStack

- 正規化した技術名を持つ。
- InternProfileとJobPostingの双方から利用する。

### Conversation / Message

- Conversationは企業担当Userとインターン生Userの組み合わせを表す。
- MessageはConversationと送信者Userに属し、本文、種類、送信日時を持つ。

### JobPosting / Application

- JobPostingはCompanyに属し、募集内容と公開状態を持つ。
- ApplicationはJobPosting、応募者User、Conversationに関連する。
- JobPostingと応募者の組み合わせを一意にする。

物理設計では外部キー、一意制約、検索用index、削除・匿名化方針を明示する。

## 9. 非機能要件

### 9.1 セキュリティ

- 公開環境ではHTTPSを使用する。
- CookieはHttpOnly、Secure、適切なSameSite属性を設定する。
- CSRF、XSS、SQL injectionなど一般的なWeb脆弱性を考慮する。
- 認証・認可と入力検証はAPI側で実施する。
- 秘密情報は環境変数で管理し、Gitへ登録しない。
- 個人情報、認証情報、メッセージ本文を不要にログへ出力しない。

### 9.2 性能

- インターン生一覧は20件単位で取得する。
- N+1クエリを避け、検索対象へ適切なindexを設定する。
- 開発用データ量で主要画面が実用的な時間内に表示されること。
- 専用検索エンジン、Redis、WebSocketは使用しない。

### 9.3 再現性・運用

- セットアップ手順をREADMEに記載する。
- migrationと架空のseedデータでデモ環境を再構築できる。
- APIにヘルスチェックを設ける。
- GitHub Actionsでテスト、lint、buildを実行する。
- 無料デモ環境の停止・有効期限などの制約をREADMEへ明記する。

### 9.4 ユーザビリティ・アクセシビリティ

- PCとスマートフォンの基本的な画面幅で操作できる。
- キーボードで主要操作を行える。
- 入力欄にはlabelを設け、エラー内容を利用者へ伝える。
- 色だけに依存して状態を表現しない。
- 破壊的操作と応募実行前に確認を表示する。

## 10. 技術・公開要件

- フロントエンド: Next.js / TypeScript
- バックエンド: Rails JSON API
- データベース: PostgreSQL
- API: REST、`/api/v1`
- 認証: HttpOnly Cookieによるセッション方式
- ローカルDB: Dockerで起動
- フロントエンド公開先: Vercel無料構成
- Rails API・PostgreSQL公開先: Render無料構成
- 公開URLはVercel側へ統一し、Rails APIをリバースプロキシする。
- ソースコードはGitHubで管理し、主要判断とセットアップ方法を文書化する。

Render無料構成では、Rails APIのアイドル停止とPostgreSQLの有効期限がある。
デモ前に起動確認を行い、データ保持を保証する環境として扱わない。

## 11. MVP受け入れ条件

### AC-01 インターン生登録・掲載

Given 未登録の利用者がインターン生として登録する  
When 必須プロフィールを保存する  
Then アカウントとプロフィールが作成され、企業向け一覧へ掲載される。

### AC-02 企業登録・ログイン

Given 未登録の利用者が企業担当者として登録する  
When 企業名、メールアドレス、パスワードを正しく入力する  
Then 企業担当者としてログインし、インターン生一覧を閲覧できる。

### AC-03 インターン生検索

Given 企業担当者がログインしている  
When 学校名、希望職種、技術スタックで絞り込む  
Then 条件をすべて満たす掲載済みインターン生が20人単位で表示される。

### AC-04 スカウト送信

Given 企業担当者がインターン生詳細を閲覧している  
When 有効な本文を送信する  
Then 会話が作成または再利用され、メッセージが履歴へ表示される。

### AC-05 メッセージ確認・返信

Given メッセージを受信した利用者がログインしている  
When 対象会話を開いて返信する  
Then 双方が送信日時順の履歴を閲覧できる。

### AC-06 募集掲載

Given 企業担当者がログインしている  
When 自社募集を作成して公開する  
Then インターン生が公開募集一覧と詳細を閲覧できる。

### AC-07 募集応募

Given インターン生が公開募集の詳細を閲覧している  
When 確認後に応募する  
Then 応募が1件作成され、会話に応募メッセージが自動送信される。

### AC-08 重複応募防止

Given インターン生が対象企業の募集へ応募済みである
When 同じ企業の別募集へ応募しようとする
Then 新しい応募と自動メッセージは作成されず、応募済みと表示される。

### AC-09 アクセス制御

Given リソースの閲覧権限を持たない利用者が存在する  
When URLまたはAPIを直接指定する  
Then プロフィール、会話、非公開募集、応募内容は開示されない。

### AC-10 入力エラー

Given 利用者が入力画面を開いている  
When 必須項目が空、上限超過、または不正な値で送信する  
Then 不正なデータは保存されず、修正方法が分かるエラーが表示される。

### AC-11 アカウント削除

Given 利用者が削除内容を確認している  
When アカウント削除を確定する  
Then ログインと新規操作ができなくなり、相手側の既存履歴には
「退会済みユーザー」としてメッセージが残る。

### AC-12 公開デモ

Given 公開環境へ最新版がデプロイされている  
When Vercelの公開URLへアクセスする  
Then HTTPSで主要導線をデモできる。
